# 1.概述 

- 地面站支持MAVLINK协议双向转发。

即把操控端的MAVLINK数据转发给无人机，实现控制无人机的目的。

操控端可以通过解析飞机的MAVLINK消息来获取飞机的状态。

- 地面站支持JSON格式上报无人机状态，但不支持JSON协议控制无人机。

- 协议支持UDP和TCP。

- 多机控制时，每台无人机ID需设置不一样。

# 2.飞控MAVLINK协议格式

飞控采用开源的mavlink2.0协议。\
飞控默认上电是mavlink1.0协议，需发送MAVLINK2.0心跳包(节2.5.2)给飞控，飞控会自动切换成mavlink2.0。\
<https://mavlink.io/zh/guide/serialization.html>

（6）mavlinkV2 的msgid 字段占3个字节，小端序。

> 属于额外校验字节，每个报文不一样。

（9）某些报文MavlinkV2发送的PAYLOAD会比MAVLINK1多(拓展)，例如报文147、69等。

## 2.3 Mavlink校验算法

uint16_t checksum= 0xffff;

For(int i=0; i\<len;i++){

crc_accumulate (buffer\[i\], &checksum);

}

crc_accumulate ([CRC_EXTRA]{.mark}, &checksum);

void crc_accumulate(uint8_t data, uint16_t \*crcAccum)

{

uint8_t tmp;

tmp = data \^ (uint8_t)(\*crcAccum &0xff);

tmp \^= (tmp\<\<4);

\*crcAccum = (\*crcAccum\>\>8) \^ (tmp\<\<8) \^ (tmp \<\<3) \^
(tmp\>\>4);

}

如上，校验需最后包含一个额外校验字节CRC_EXTRA，校验算法:
CRC-16/MCRF4XX，C/C++具体算法请看文件[MAVLINK校验算法.txt]{.mark}

### 2.6.4 任务执行流程

如果是电子围栏，上传后自动生效。触碰围栏无人机默认悬停，需往回控制飞机，飞出围栏。无人机在禁飞区外不允许解锁

当无人机在地面

- 检查无人机的定位状态，确保GPS可用。

- 上传航线到无人机。

- 切换无人机模式到任务模式，然后发送解锁指令。

当无人机在空中

- 检查无人机的定位状态，确保GPS可用。

- 上传航线到无人机。

- 切换无人机模式到任务模式。

暂停任务

- 可切换飞行模式为定点模式

- 使用摇杆(舵量不在中位)来夺取控制权

继续执行任务

- 切换回任务模式即可从断点处继续执行任务。

- 重新上传航线再切换回任务模式，航线会重头开始执行。

## 2.9 发送RTCM差分数据给无人机

+----------------------+----------------------+------------------+------------------+
| 报文MAVLINK_MSG_ID_GPS_RTCM_DATA，ID=233    | 载荷长度:                           |
|                                             | 182，数据流向：操控端-\>飞机        |
+----------------------+----------------------+------------------+------------------+
| 简写                 | 类型                 | 偏移量           | 说明             |
+----------------------+----------------------+------------------+------------------+
| flags                | uint8_t              | 0                | 标志位           |
+----------------------+----------------------+------------------+------------------+
| len                  | uint8_t              | 1                | 数据长度         |
+----------------------+----------------------+------------------+------------------+
| Data\[180\]          | uint8_t              | 2                | 差分数据         |
+----------------------+----------------------+------------------+------------------+

flags标志位说明:

- 位0：
  当一帧差分数据小于180字节时(即一个报文可发完一帧差分数据)，位0为0(Bit0
  = 0)，否则为1(Bit0 = 1)，

- 位1-2：当一帧差分数据大于180字节时，则需多次报文(最多4个)才能发送完整的一帧差分数据，位2-3表示的是这个4个报文的序号SUBSEQ，即\[0,1,2,3\]。

- 位3-7：表示的是完整的一帧差分数据的报文序号SEQ，每帧SEQ+=1。

假设现在有3帧差分数据，第一帧总字节数是550，那么需要发送4次报文才能发送完，

第一次发送180字节，剩余370字节，Bit0 = 1, SUBSEQ = 0，SEQ=0

第二次发送180字节，剩余190字节，Bit0 = 1, SUBSEQ = 1，SEQ=0

第三次发送180字节，剩余10字节， Bit0 = 1, SUBSEQ = 2，SEQ=0

第三次发送10字节，剩余0字节， Bit0 = 1, SUBSEQ = 3，SEQ=0

第二帧差分数据总字节数是200，需要发送4次报文才能发送完，

第一次发送180字节，剩余20字节，Bit0 = 1, SUBSEQ = 0，SEQ=1

第一次发送20字节，剩余0字节， Bit0 = 1, SUBSEQ = 1，SEQ=1

第三帧差分数据总字节数是100，发送1次报文就能发送完，

第一次发送100字节，剩余0字节， Bit0 = 0, SUBSEQ = 0，SEQ=2
