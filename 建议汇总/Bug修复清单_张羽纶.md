# “云翼智控”网站 Bug 处理清单

### 1. 备份任务状态卡死
*   **问题描述**：所有的备份任务都卡死在“进行中”状态，且文件大小显示为 0 B，耗时为 0ms。
*   **复现步骤**：
    1. 登录系统，点击左侧菜单“系统调度与运维” -> “备份与数据回滚”。
    2. 查看下方的“备份记录”表格。
*   **解决提示词 (给开发/AI)**：
    > **【模块定位】**：后端系统调度服务 / 数据库备份模块
    > **【问题成因】**：备份任务是由异步 Worker 或定时任务触发的，任务提交后状态变为“进行中”，但在执行过程中可能因为权限不足、内存溢出或数据库锁死导致进程崩溃；且代码中缺乏 `try-catch` 异常捕获和状态回调机制，导致数据库中的任务状态永远无法被更新为“成功”或“失败”。
    > **【修复建议】**：
    > 1. 排查执行数据备份的底层异步任务（如 Celery, Quartz 或自定义 Goroutine）的执行日志，修复导致崩溃的核心异常。
    > 2. 完善任务回调逻辑：确保任务执行无论成功还是抛出异常，都会触发 finally 块去更新数据库表中的 `status`、`file_size` 和 `duration` 字段。
    > 3. 增加超时兜底机制：对于处于“进行中”状态超过设定时间（如 1 小时）的任务，由一个独立的清理脚本自动标记为“超时失败”。

### 2. 电池监控统计数据脱节
*   **问题描述**：电池监控页面顶部的数据汇总卡片全部显示为 0，与下方表格中实时渲染的在线设备数据脱节。
*   **复现步骤**：
    1. 点击左侧菜单“无人机状态监控” -> “电池监控”。
    2. 对比页面顶部的统计卡片与下方的“电池实时监控”表格中的实际无人机数量。
*   **解决提示词 (给开发/AI)**：
    > **【模块定位】**：前端页面渲染 / “电池监控”视图组件
    > **【问题成因】**：页面顶部的数据汇总卡片与下方的实时监控表格使用了两套独立的数据源或状态。可能是后端接口返回统计数据的结构发生变更（例如增加了一层 `data` 嵌套），导致前端解构赋值失败并回退到默认值 `0`；或者由于组件生命周期问题，统计卡片在表格数据拉取前就已渲染且未建立响应式依赖。
    > **【修复建议】**：
    > 1. 检查该页面的数据获取 API（如有独立的统计接口，使用控制台 Network 查看返回值格式是否与前端 Model 匹配）。
    > 2. 推荐的终极修复方案：如果表格本身已经拉取了所有设备的详细数据，应当直接废弃统计接口，在前端 Vue/React 组件中编写 Computed 属性（如计算数组长度），动态生成顶部卡片的汇总数值，从根本上杜绝数据脱节。

### 3. CoT 智能决策时间解析错误
*   **问题描述**：CoT 智能决策详情模态框中，“创建时间”字段显示为无效的默认时间 `1/01/01 08:00`。
*   **复现步骤**：
    1. 点击左侧菜单“仿真与智能决策” -> “CoT 智能决策”。
    2. 点击推理记录列表中的任意“查看”按钮，查看弹出框中的“创建时间”字段。
*   **解决提示词 (给开发/AI)**：
    > **【模块定位】**：前端 UI 组件库 / “CoT 智能决策”详情模态框
    > **【问题成因】**：显示为 `1/01/01 08:00` 属于典型的 Unix Epoch 零值时间（在东八区加上 8 小时偏移）。这通常是因为后端传递的 `创建时间` 字段为空（null/0）、字段名拼写错误（如把 `createdAt` 写成了 `create_time`），或者是前端的时间格式化函数（如 Day.js, Moment）接收到了无法解析的字符串，从而将其处理为默认初始时间。
    > **【修复建议】**：
    > 1. 抓包核对当前详情接口返回的 JSON 数据，确认时间字段的键名和值（是秒级时间戳、毫秒级时间戳还是 ISO 字符串）。
    > 2. 检查前端模态框组件绑定的字段名是否与后端一致。
    > 3. 更新前端时间解析 Utils，增加判空逻辑：若后端返回空值，则渲染为 `-` 或 `未知时间`，防止出现引起误解的默认零值时间。

### 4. 异常报警产生“告警风暴”
*   **问题描述**：异常报警模块缺乏去重逻辑，同一设备短时间内产生大量相同报警记录，引发“告警风暴”。
*   **复现步骤**：
    1. 登录系统，进入“异常报警”页面。
    2. 观察同一台无人机在短时间（如 1 分钟）内的告警生成频率，会发现数十条完全相同的记录。
*   **解决提示词 (给开发/AI)**：
    > **【模块定位】**：后端告警服务 / 消息中间件消费端
    > **【问题成因】**：无人机在发生异常时（如电量低），其传感器会以极高的频率上报状态。后端的报警处理模块仅作了“透传”，没有在业务层加入“状态缓存”和“过滤拦截”机制，导致每一次传感器的异常上报都被当作独立事件写入数据库，最终造成数据库雪崩和前端页面满屏冗余信息。
    > **【修复建议】**：
    > 1. 在告警生成逻辑前置引入 Redis 缓存。对于每一台设备的特定告警类型，设置一个带有 TTL（如 5 分钟）的 Key 作为“静默期（Cooldown）锁”。
    > 2. 若发现锁已存在（说明该设备此问题正在告警中），则拒绝生成新的数据库记录，而是执行 `UPDATE` 操作去累加上一条未解决告警的 `发生次数 (count)` 并更新 `最近上报时间`。
    > 3. 只有当锁过期，或者运维人员手动点击了“处理完毕”清除了该状态，才允许针对同一问题生成全新的告警记录。

### 5. AI 助手面板遮挡操作列
*   **问题描述**：右下角的 AI 助手悬浮面板展开时，遮挡了数据表格最右侧的操作列，影响业务点击。
*   **复现步骤**：
    1. 在“电池监控”等带有数据表格的页面，点击右下角唤起 AI 助手。
    2. 尝试操作表格最右侧的一列（详情、修改等按钮），发现被面板遮挡。
*   **解决提示词 (给开发/AI)**：
    > **【模块定位】**：前端全局布局组件 (Layout) / CSS 样式
    > **【问题成因】**：右下角的“AI 助手”组件使用了脱离标准文档流的绝对定位或固定定位（`position: fixed/absolute`），且层级（z-index）较高。而在页面排版时，并未给这个预期的悬浮面板预留空间，导致当它展开时，直接悬浮在底部的 DOM 元素（即数据表格最右侧的操作按钮）之上，发生物理遮挡。
    > **【修复建议】**：
    > 1. **方案 A（响应式挤压）**：监听 AI 助手的 `isExpanded` 状态，如果展开，则给主页面容器（如 `.main-content`）动态添加一个对应的 `padding-right`（如 400px），迫使表格内容自适应收缩，不被遮挡。
    > 2. **方案 B（表格内部适配）**：修改该页面的表格组件样式，确保表格父容器设置了 `overflow-x: auto;` 并且设定了最大宽度。同时，如果右侧“操作列”设置了 `position: sticky`，请调整它的 `right` 偏移量，使其在助手展开时能被正常滚动查看。
