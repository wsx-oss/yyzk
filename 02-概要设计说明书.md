## 一、项目介绍

### 1.1 项目名称

云翼智控—— 基于强化学习与 RAG 的无人机群智能管控平台

### 1.2 项目背景

随着低空经济政策加速落地与无人机规模化应用场景的持续拓展，无人机集群在电力巡检、地理测绘、物流配送、应急搜救等行业的部署密度显著增长。然而，当前无人机管控体系普遍存在三类突出问题：一是**多机状态分散，缺乏统一监管**，无人机运行状态与位置信息分散，难以实现多机统一可视化监测与集中管理；二是**异常处置依赖经验，智能分析不足**，面对飞行异常和告警事件，传统方式高度依赖人工经验，缺乏知识增强与智能辅助分析能力；三是**跨网接入复杂，远程运维成本高**，无人机部署环境复杂，跨网络数据接入、远程调试与运维排障流程繁琐，协同效率较低。

在此背景下，开发一套面向具身智能无人系统的人机交互与远程智控平台，构建"边缘感知—云端计算—前端展示"三层贯通架构，融合大语言模型（LLM）、检索增强生成（RAG）、案例推理（CBR）与强化学习（RL）等前沿智能技术，形成覆盖"任务规划—执行监控—策略优化—安全回滚"全链路的智能管控能力，成为当前无人机领域迫切的工程需求。

### 1.3 项目意义

本项目的意义主要体现在以下三个层面：

 **（1）学术探索价值。** 项目探索将大模型推理与强化学习策略引入无人机任务规划与异常决策领域，构建了 CoT 思维链可解释分析与 RAG 领域知识检索增强的协同机制，为具身智能体的自主决策可解释性研究提供了可参考的自主可控工程范式。

 **（2）技术创新价值。** 在技术层面，项目实现了 LLM 与 NFZ 几何纠偏的双引擎航线规划、多实例仿真与 Q-Learning 策略验证闭环、禁飞区地理围栏与防碰撞联动等关键能力，形成了从仿真训练到实机部署的 sim-to-real 策略迁移通路。同时，系统引入 CBR 案例推理机制，可自动检索历史相似飞行案例为当前决策提供经验参考，增强了智能规划的上下文感知能力。

 **（3）工程实践价值。** 在工程层面，项目建设了完整覆盖任务全流程的 20 个业务模块、34 张数据库表、统一并发任务框架（IO/CPU 双池调度）与全量数据持久化体系，具备即开即用的工程化交付能力与良好的可扩展性。

### 1.4 项目发展历程

在项目立项阶段，团队围绕“基于强化学习与 RAG 的无人机群智能管控”核心场景开展需求调研与技术选型论证，最终确定以 Go 语言 + Gin 框架 + MySQL/SQLite 双驱动为核心技术路线，面向无人机集群管控场景进行平台化建设。

第一阶段（2026 年 1 月）：项目立项与基础架构搭建。完成需求分析与总体方案设计，确定 Go 1.25 + Gin + SQLite/MySQL 双驱动技术路线。同步完成用户认证体系、基础路由与中间件框架建设，形成无人机管理、GPS 地图展示、电池监控等首批功能原型。

第二阶段（2026 年 2 月）：核心业务模块开发与智能能力初步集成。完成飞行任务两步向导、禁飞区管理与 NFZ 纠偏引擎、LLM 智能航线规划、CoT 思维链分析、SSH/VNC/RDP 远程控制、异常报警、维护日志、性能分析等主体模块建设，并建立 WebSocket 统一事件广播中心。

第三阶段（2026 年 3 月上旬）：并发框架建设与平台服务化能力提升。完成 MySQL 生产级迁移、统一并发框架（TaskPool）建设、高频数据批写与推送节流优化、备份与数据回滚、AI 智能助手与消息通知中心、并发监控仪表盘等关键能力建设。

第四阶段（2026 年 3 月下旬）：仿真引擎与数据全量持久化。完成多实例仿真引擎、Q-Learning 强化学习训练链路、全量数据持久化升级（34 张表）、Redis 缓存集成、贝塞尔曲线与 Catmull-Rom 样条轨迹平滑等核心建设。

第五阶段（2026 年 4 月）：智能能力增强与阶段性成果整合。完成 RAG 检索增强生成正式集成、LLM RPM 令牌桶限流、离线状态通知抑制、天地图体系迁移、前端毛玻璃视觉效果建设、RL 策略导出与任务联动等优化。截至目前，平台已形成覆盖“感知—管控—分析—决策—仿真—训练”的完整能力链条。

## 二、预期设计

### 2.1 设计目标

在基于强化学习与 RAG 的无人机群智能管控平台项目中，设计目标中的关键质量属性是确保系统性能、用户体验和市场竞争力的核心要素。以下是针对该项目设计目标的详细描述：

#### 2.1.1 高并发管控能力

定义：系统能够同时管控大规模无人机集群，支持多实例并行运行与实时状态同步。

量化指标：并发管控无人机数量 ≥ 200 台；仿真引擎支持 200 台实例 7×24 连续运行；Worker Pool 任务队列容量 1024，IO 池 16 线程 + CPU 池 4 线程。

重要性：高并发管控能力是平台核心竞争力的基础，直接决定了系统在实际部署场景（如电力巡检、应急搜救）中的可用性和可靠性。

#### 2.1.2 低延迟实时推送

定义：系统能够快速处理并推送实时遥测数据，确保前端展示与实际状态的时间差最小化。

量化指标：WebSocket 端到端推送延迟 ≤ 200ms（局域网环境）；GPS 推送节流间隔 200ms，电池推送节流间隔 500ms；系统从启动到进入工作状态的时间 ≤ 5 秒。

重要性：低延迟实时推送是保障操作人员及时掌握无人机状态、实施精准决策的关键前提，尤其在紧急避障与任务调度场景中至关重要。

#### 2.1.3 智能决策辅助

定义：系统能够基于大语言模型、检索增强生成、案例推理与强化学习等技术，为用户提供智能化的航线规划、故障诊断与策略优化能力。

量化指标：LLM 航线规划支持多候选方案并行生成；RAG 知识库覆盖 20 篇领域文档；CoT 思维链推理覆盖 6 类分析场景；RL 策略训练 ≤ 1000 episode 初步收敛。

重要性：智能决策辅助能力是系统区别于传统管控平台的核心差异化优势，直接提升了平台在复杂动态环境下的自适应能力与用户决策效率。

#### 2.1.4 高可靠性

定义：系统在各种环境条件下能够稳定运行，且故障率低。

量化指标：系统支持 7×24 连续运行无崩溃；Panic 自动恢复；Redis 不可用时自动降级为内存模式；LLM 不可用时自动降级为直线规划。

重要性：高可靠性确保平台在长时间使用中不会频繁出现故障，降低维护成本。稳定的系统性能能够提升用户对产品的信任度，增强市场竞争力。

#### 2.1.5 用户友好性

定义：系统操作界面简洁直观，用户无需复杂培训即可使用，且人机交互自然流畅。

量化指标：用户首次使用的学习时间 ≤ 5 分钟；20 个功能模块通过统一仪表盘导航；AI 智能助手支持自然语言交互与模块快速跳转。

重要性：用户友好性直接影响产品的市场接受度和用户体验。简单易用的操作方式能够吸引更多非专业用户，扩大市场范围。

#### 2.1.6 安全性

定义：系统在运行过程中能够确保用户和无人机的安全，避免碰撞或意外事故。

量化指标：仿真引擎内置防碰撞检测与自动航向/高度避让；禁飞区地理围栏 Ray-Casting 实时检测；API 限流 3000 次/分钟/IP（Redis 分布式 + 内存降级）；用户密码 bcrypt 加密存储，Token 24h 过期。

重要性：安全性是产品的基本要求，尤其是在无人机飞行环境中使用时。高安全性能能够减少事故风险，保护用户和设备的安全。

#### 2.1.7 可扩展性

定义：系统能够方便地进行功能扩展或组件升级，以适应未来的需求变化。

量化指标：数据库支持 SQLite 与 MySQL 双驱动无缝切换；AdaptSQL 兼容层自动处理 SQL 方言差异；Go embed 单二进制部署，支持交叉编译。

重要性：可扩展性确保产品在未来能够适应新的应用场景或技术升级，提高产品的生命周期和市场竞争力。

#### 2.1.8 成本效益

定义：在保证性能的前提下，系统的设计和部署成本应具有市场竞争力。

量化指标：单体部署二进制文件体积 ≤ 50 MB（Go embed 内嵌前端静态资源）；无需安装 Node.js 或前端构建工具；配合 `.env` 文件即可独立部署运行。

重要性：成本效益直接影响产品的市场定价和市场接受度。在保证性能的同时控制成本，能够提高产品的市场竞争力和经济效益。

以上关键质量属性是本项目的核心设计目标，涵盖了并发能力、响应速度、智能化程度、可靠性、安全性和成本效益等关键方面。这些属性不仅确保了产品的高性能和用户体验，还为项目的开发和市场推广提供了明确的指导和评估标准。

### 2.2 设计原则

以下是该项目在性能、可靠性、用户体验和成本效益等方面的设计原则：

#### 2.2.1 用户中心原则

易用性：操作界面简洁直观，20 个功能模块通过统一仪表盘侧边栏导航，用户无需复杂培训即可上手。

舒适性：在设计中考虑用户的使用习惯，例如 AI 智能助手浮窗支持拖拽移动与位置持久化，通知铃铛支持类型筛选与批量已读管理，可根据用户偏好调整。

反馈机制：通过 WebSocket 实时推送与前端可视化图表，持续向用户反馈系统状态与任务进度，形成信息闭环。

#### 2.2.2 可靠性原则

冗余设计：关键服务采用多层降级策略，Redis 不可用时自动降级为纯内存模式，LLM 不可用时降级为直线规划，避免单点故障导致系统不可用。

容错机制：系统能够自动检测并处理异常情况。Panic 自动恢复机制确保服务进程不会因单个请求异常而崩溃；仿真实例低电量自动检测与返航机制保障仿真连续性。

测试验证：通过严格的测试流程，包括 9 项并发压测用例、单元测试和集成测试，确保系统的可靠性。

#### 2.2.3 安全性原则

认证与授权：完善的用户认证体系（bcrypt 密码加密 + Token 会话管理），所有受保护接口均需有效 Token 鉴权，过期 Token 返回 401 Unauthorized。

限流防护：采用 Redis 分布式限流 + 内存降级双重机制（3000 次/分钟/IP），有效防止恶意请求与 DDoS 攻击。

飞行安全：内置禁飞区地理围栏管控、防碰撞检测与自动避让机制，确保无人机在飞行过程中不会进入禁飞区域或发生碰撞事故。

数据安全：全量数据持久化至数据库，备份恢复前自动创建安全备份，操作日志完整记录审计链路。

#### 2.2.4 模块化设计原则

后端模块化：采用分层架构设计，将 API 处理（handlers）、业务逻辑（internal）、数据访问（db）、并发调度（taskpool）等封装为独立模块，便于代码复用和功能扩展。

前端模块化：20 个功能模块以独立 HTML 页面形式组织，通过 iframe 承载于统一仪表盘框架，抽离 `common.js`、`common.css` 公共资源，确保模块间解耦与统一样式基线。

接口标准化：定义清晰的 RESTful API 接口规范（统一分页参数、响应格式），确保前后端模块之间的兼容性和互操作性。

#### 2.2.5 可扩展性原则

数据库可扩展：通过 AdaptSQL 兼容层实现 SQLite/MySQL 双驱动无缝切换，单套代码兼容本地开发与云端生产环境，预留向更高性能数据库迁移的工程通道。

功能可扩展：采用模块化 Handler 注册模式，新增功能模块仅需添加路由注册与 Handler 实现，无需修改核心框架代码。

部署可扩展：全量数据存储于数据库（消除本地文件依赖），支持多实例水平扩展与云端无状态部署。

#### 2.2.6 性能优化原则

算法优化：采用 BM25 关键词检索算法实现 RAG 知识库高效检索，Q-Learning 强化学习算法实现策略训练收敛，WriteBatcher 批量写入减少数据库 I/O 压力。

并发优化：统一 Worker Pool 双池架构（IO 16 线程 + CPU 4 线程）配合 4 级优先级队列，实现后台任务高效调度。Throttler 节流器控制 WebSocket 推送频率，避免消息风暴。

实时性设计：确保系统能够实时处理数据并快速响应，通过 WebSocket 事件驱动推送机制与统计缓存异步刷新策略提升用户体验。

#### 2.2.7 成本效益原则

经济性选型：在满足功能需求的情况下，优先选择成本较低的开源技术组件（Go、Gin、MySQL、Redis、Leaflet.js、Chart.js）。

开源软件：使用开源软件和工具，降低开发成本。Go 语言标准库覆盖面广，减少第三方依赖数量。

部署友好性：设计易于部署的单二进制交付方式（Go embed 内嵌前端静态资源），一个可执行文件 + 一个 `.env` 配置文件即可运行全部功能，降低运维成本。

以上设计原则是本项目的核心指导思想，涵盖了用户中心、可靠性、安全性、模块化、可扩展性、性能优化、成本效益等方面。这些原则不仅确保了产品的高性能和用户体验，还为项目的开发和市场推广提供了明确的方向和约束，帮助团队在复杂多变的开发过程中保持一致性和高效性。

### 2.3 设计决策

本项目是基于无人机群智能管控场景的 Web 平台项目，主要使用 Go 语言编程，结合 Gin Web 框架实现模块化开发和通信。代码中主要使用了 RESTful API 与 WebSocket 实现前后端交互和实时数据推送。

后端服务基于 Go 1.25 + Gin 框架构建，选择 Go 语言的核心原因在于其天然的高并发能力（goroutine 轻量级协程）、优秀的编译性能（单二进制交付）与丰富的标准库支持。Gin 框架具备高性能路由匹配、中间件链式处理与 JSON 序列化优化能力，适合构建 RESTful API 密集型应用。

数据层采用 MySQL 作为主存储引擎（支持 SQLite 降级），通过 AdaptSQL 兼容层实现 SQL 方言自动适配。Redis 作为缓存中间件承担会话加速、分布式限流、统计同步与通知缓存职能。这种算法组合兼具较高的实时性和一定的精度。

智能决策层融合 LLM 大语言模型（航线规划与推理）、BM25 RAG 检索引擎（知识增强）、CoT 思维链（可解释分析）、CBR 案例推理（经验复用）与 Q-Learning 强化学习（策略训练），形成多智能技术协同的决策架构。

前端采用原生 HTML5/CSS3/JavaScript 开发，集成 Leaflet.js 地图引擎与 Chart.js 图表库，通过 Go embed 机制将静态资源嵌入后端二进制文件，实现一体化交付。

硬件开发涉及无人机飞控（MAVLink 协议）与 4G 通信模块（遥测数据直传云端）等组件，开发工具包括 Go 编译环境与标准 Web 开发工具链。

## 三、系统架构和整体设计

### 3.1 系统架构

#### 3.1.1 硬件架构

本系统采用"边缘感知—云端计算—前端展示"三层贯通的硬件体系架构，各层职责明确、接口清晰。

**（1）边缘感知层——无人机平台。** 由多旋翼无人机集群构成，每架无人机搭载飞控系统（支持 MAVLink 通信协议）、GPS/GNSS 定位模块、惯性测量单元、电池管理单元（BMS) 与任务载荷 。飞控系统负责接收控制指令并执行飞行动作，GPS 模块以≥1 Hz上报经纬度与高度信息，BMS 模块周期性上报电压、电流、电量百分比与温度数据。无人机端无需运行额外程序，飞控遥测数据经机载 4G LTE 模块通过蜂窝网络直接上传至云端服务器，通信链路支持断线自动重连与心跳保活机制。

**（2）云端计算层——后端服务。** 部署于云端或本地机房的服务器集群，运行本平台核心后端服务。服务器需具备多核 CPU（≥4 核）、≥8 GB 内存、≥100 GB 存储空间的硬件配置。后端内置 TCP Gateway 直接接收无人机 4G 上传的遥测数据并解析 MAVLink 协议，同时承载 API 网关、业务逻辑、并发调度、AI 推理等全部核心能力。数据库服务器运行 MySQL 8.x 实例（开发环境可降级为 SQLite），Redis 实例承担缓存与限流职能。

**（3）前端展示层——用户终端。** 操作人员通过标准浏览器访问 Web 管理界面，无需安装任何客户端软件。支持 PC 端全功能操作与平板端基础监控，通过 WebSocket 长连接接收实时数据推送。

#### 3.1.2 硬件详细设计

**（1）无人机端硬件设计。** 飞控系统采用飞控，通过 MAVLink v2 协议输出遥测数据。GPS 模块采用双频 RTK 定位方案，定位精度厘米级。电池管理系统支持多芯串联电池组的实时监测，BMS 数据通过飞控串口转发至 4G 通信模块。无人机端无需搭载额外计算设备或运行任何程序，所有遥测数据经 4G 模块直接上传至云端服务器进行处理。

**（2）通信模块设计。** 4G 通信模块采用工业级 LTE Cat.4 模组，支持 TCP/UDP 双协议栈。数据上行带宽 ≥5 Mbps，下行带宽 ≥10 Mbps，满足遥测数据传输与远程桌面流媒体需求。通信模块内置 SIM 卡槽与外置天线接口，支持运营商 APN 配置与专网接入。

**（3）服务器硬件需求。** 生产环境推荐配置：CPU ≥4 核（推荐 8 核）、内存 ≥8 GB（推荐 16 GB）、SSD 存储 ≥100 GB、网络带宽 ≥100 Mbps。MySQL 与 Redis 服务可部署于同一物理节点或独立节点，根据并发规模灵活调整。

#### 3.1.3 软件架构设计

本系统后端采用经典的分层架构，自上而下分为：

**（1）路由与中间件层（Router & Middleware）。** 基于 Gin 框架构建 RESTful API 路由树，挂载认证中间件（Token 鉴权）、限流中间件（Redis/内存双模式）、CORS 跨域中间件与 Panic Recovery 中间件。所有 HTTP 请求首先经过中间件链处理后路由至对应 Handler。

**（2）Handler 处理层。** 按业务领域划分为多个 Handler 文件（api.go、drones.go、gps.go、battery.go、flight.go、ai_assistant.go、cot.go、sim.go、rl.go 等），每个 Handler 负责参数解析、业务逻辑调用与响应序列化。Handler 之间保持松耦合，通过共享的 DB 句柄与 WSHub 实例实现数据共享。

**（3）业务逻辑层（Internal Packages）。** 包含核心业务组件：`internal/taskpool` 提供统一并发任务框架（IO/CPU 双池调度、4 级优先级队列、批量执行器与节流器）；`internal/agent` 集成智能决策引擎（LLM 航线规划、CoT 思维链分析、CBR 案例推理与 RAG 检索增强）；`internal/sim` 实现仿真模拟引擎（多实例并行仿真、碰撞检测、地理围栏校验与状态快照）；`internal/rl` 实现强化学习训练模块（Q-Learning 算法、奖励函数计算与策略导出）；`internal/cache` 提供统计缓存层（异步定时刷新机制提供全局/分设备统计数据）。

**（4）数据访问层（DB Wrapper）。** 自研 DB 封装层支持 SQLite 与 MySQL 双驱动，通过 `AdaptSQL` 函数自动将 SQLite 方言（如 `INSERT OR REPLACE`）转换为 MySQL 方言（如 `REPLACE INTO`）。数据库迁移采用 `internal/db/migration.go` 统一管理 34 张业务表的 DDL 语句。

**（5）实时通信层（WSHub）。** 基于 Gorilla WebSocket 库构建的 Hub-Client 广播架构，支持连接注册/注销、房间隔离、节流广播（ThrottledBroadcast）与消息类型过滤。前端通过 WebSocket 长连接订阅 GPS 更新、电池变化、任务状态、告警通知等实时事件。

#### 3.1.4 关键算法设计

**（1）LLM + NFZ 双引擎航线规划算法。** 用户输入起点、终点与约束条件后，系统首先调用 LLM API 生成初始航线方案（JSON 格式的航点序列），随后由 NFZ 几何纠偏引擎对航线进行禁飞区合规性校验。若航线穿越禁飞区，引擎通过 Ray-Casting 算法检测冲突航段，并基于禁飞区边界几何信息自动生成绕飞航点。纠偏后的航线再经贝塞尔曲线平滑处理，最终输出安全可执行的飞行路径。

**（2）BM25 RAG 检索增强生成算法。** 知识库文档在加载时执行预处理：分词、去停用词、计算 IDF 权重。用户查询时，系统对查询文本执行相同的分词处理，然后使用 BM25 评分公式计算查询与每篇文档的相关性得分。取 Top-K（K=3）相关文档作为上下文注入 LLM Prompt，实现基于领域知识的增强回答。

**（3）Q-Learning 强化学习算法。** 仿真环境定义状态空间（无人机位置、电量、距目标距离）与动作空间（前进、转向、悬停、返航），奖励函数综合考量距离进展、电量消耗、碰撞惩罚与到达奖励。训练过程采用 ε-greedy 探索策略，ε 从 1.0 线性衰减至 0.1。Q 表以 JSON 格式持久化至数据库，支持策略导出与加载。

**（4）CoT 思维链推理算法。** 系统为 6 类分析场景（异常诊断、性能评估、航线评价、风险预警、行为分析、效率优化）预设结构化 Prompt 模板。LLM 在推理过程中逐步输出推理链条，每步包含推理依据、中间结论与置信度评估，最终生成可追溯的分析报告。

**（5）CBR 案例推理算法。** 系统维护历史飞行案例库，当新任务触发时，CBR 引擎基于特征向量相似度检索最相似的历史案例（包含环境条件、任务参数、执行结果与经验教训），将匹配案例作为上下文注入 LLM 决策 Prompt，实现经验驱动的智能决策增强。

### 3.2 总体设计

#### 3.2.1 系统初始化

系统启动时执行以下初始化序列：

**Step 1 —— 环境配置加载。** 从 `.env` 文件读取数据库驱动类型（`SC_DB_DRIVER`）、数据库连接字符串（`SC_DB_DSN`）、Redis 连接参数（`REDIS_ADDR`/`REDIS_PASSWORD`/`REDIS_DB`）、LLM API 密钥与端点、服务监听端口等配置项。

**Step 2 —— 数据库初始化。** 根据驱动类型打开 SQLite 或 MySQL 连接，执行 `migrate.AutoMigrate()` 自动创建/更新 34 张业务表结构。数据库连接池配置最大打开连接数与最大空闲连接数。

**Step 3 —— Redis 初始化。** 尝试连接 Redis 实例并执行 PING 测试。若连接失败，系统打印警告日志并自动降级为纯内存模式，不影响核心功能运行。

**Step 4 —— 并发框架启动。** 初始化 TaskPool，启动 IO Worker Pool（16 goroutine）与 CPU Worker Pool（4 goroutine），注册周期性后台任务（统计缓存刷新、WriteBatcher 定时 Flush 等）。

**Step 5 —— WebSocket Hub 启动。** 初始化 WSHub 实例，启动消息分发协程，等待前端客户端建立 WebSocket 连接。

**Step 6 —— 路由注册与服务启动。** 创建 Gin Engine，挂载全局中间件，按功能模块注册 API 路由与静态资源路由，启动 HTTP 服务监听。

#### 3.2.2 WebSocket 实时通信

系统采用 Hub-Client 模式实现 WebSocket 实时通信：

**Step 1 —— 连接建立。** 前端页面加载后，JavaScript 客户端向 `/ws` 端点发起 WebSocket 握手请求。Hub 接收新连接后将 Client 注册至活跃连接池。

**Step 2 —— 消息分发。** 后端业务模块（GPS Handler、Battery Handler、Flight Handler 等）在数据更新时调用 `hub.BroadcastJSON(eventType, payload)` 发布事件。Hub 遍历活跃连接池，向所有已连接客户端推送 JSON 消息。

**Step 3 —— 节流控制。** 对高频数据源（GPS 200ms、电池 500ms）启用 `ThrottledBroadcast`，在节流窗口内仅保留最新一条消息，避免消息风暴。

**Step 4 —— 断线处理。** Hub 检测到连接写入失败时自动注销该 Client，释放资源。前端实现断线自动重连机制（指数退避策略）。

#### 3.2.3 并发任务调度

统一并发框架 TaskPool 实现后台任务的高效调度：

**Step 1 —— 任务提交。** 业务模块通过 `pool.Submit(task, priority)` 将任务提交至优先级队列（Critical/High/Normal/Low 4 级）。

**Step 2 —— 队列调度。** 调度器按优先级从高到低依次取出任务，分配给空闲 Worker 执行。IO 密集型任务（数据库读写、网络请求）分配至 IO Pool，CPU 密集型任务（路径计算、策略训练）分配至 CPU Pool。

**Step 3 —— 批量执行。** WriteBatcher 组件将高频写入请求（GPS 数据、遥测记录）聚合为批次，当累积数量达到阈值或定时器触发时执行批量 INSERT，减少数据库连接开销。

**Step 4 —— 结果回调。** 任务执行完成后通过 channel 返回结果或错误，调用方可通过 Future 模式异步获取执行结果。

#### 3.2.4 智能航线规划流程

**Step 1 —— 参数收集。** 用户在前端界面输入起点坐标、终点坐标、飞行高度、速度约束与特殊要求（文本描述）。

**Step 2 —— LLM 推理。** 系统将参数组装为结构化 Prompt，调用 LLM API 生成初始航线方案。Prompt 包含系统角色设定、输出格式约束（JSON 航点数组）与安全飞行规则。

**Step 3 —— NFZ 校验与纠偏。** 对 LLM 返回的航点序列逐段执行禁飞区碰撞检测（Ray-Casting 算法）。若检测到冲突航段，自动计算禁飞区边界绕飞点并插入航线。

**Step 4 —— 轨迹平滑。** 对纠偏后的航点序列应用贝塞尔曲线平滑算法，消除急转弯航段，生成飞行器可执行的平滑轨迹。

**Step 5 —— 结果返回。** 将最终航线（航点列表、总距离、预计飞行时间）通过 API 返回前端，在 Leaflet 地图上可视化展示。

#### 3.2.5 仿真引擎运行流程

**Step 1 —— 仿真初始化。** 用户创建仿真场景，配置无人机数量（≤200）、初始位置、目标位置、电池容量、速度参数与禁飞区列表。

**Step 2 —— Tick 循环。** 仿真引擎以固定时间步长（默认 1 秒）驱动所有无人机实例的状态更新。每个 Tick 内执行：位置更新（基于速度向量）→ 电量消耗计算 → 碰撞检测 → 禁飞区校验 → 目标到达判定 → 低电量返航检测。

**Step 3 —— 碰撞避让。** 当两架无人机的欧氏距离小于安全阈值时，触发防碰撞机制：自动调整冲突方的航向偏移 30° 或高度偏移 10 米，直至安全距离恢复。

**Step 4 —— 状态广播。** 每个 Tick 结束后，引擎通过 WebSocket 向前端广播所有无人机的最新状态（位置、电量、状态码），前端实时更新地图标注与状态面板。

**Step 5 —— 快照保存。** 引擎支持定时或手动创建仿真快照，将全量状态序列化为 JSON 并持久化至数据库（`data_store` 表），支持任意时刻的状态恢复与回放。

#### 3.2.6 判断结束条件

仿真引擎根据以下条件判断仿真是否结束：

**条件 1 —— 任务完成。** 所有无人机均到达目标位置（距离 ≤ 到达判定阈值），仿真正常结束。

**条件 2 —— 电量耗尽。** 所有无人机电量降至 0%，仿真因能源耗尽结束。

**条件 3 —— 超时终止。** 仿真运行时间超过用户设定的最大时长，强制终止。

**条件 4 —— 用户干预。** 用户通过前端界面手动停止仿真或触发紧急终止。

#### 3.2.7 模块构成

本系统由 20 个功能模块构成，按职能划分为以下五大类：

**（一）基础管控类：** 用户认证模块（注册/登录/Token 会话管理）、无人机管理模块（自动发现/状态监控/属性编辑）、GPS 数据模块（实时定位/轨迹回放/地图标注）、电池监控模块（电压电流电量温度/阈值告警/趋势图表）、飞行任务模块（两步向导创建/状态流转/历史记录）。

**（二）安全管控类：** 禁飞区管理模块（区域 CRUD/地图可视化/Ray-Casting 碰撞检测）、告警管理模块（多级别告警/确认处理/实时推送）、备份与回滚模块（全量备份/一键恢复/恢复前安全备份）。

**（三）远程接入类：** 远程控制模块（SSH 终端/VNC 桌面/RDP 远程桌面三种接入方式）、视频监控模块（实时视频流查看与设备接入）。

**（四）智能决策类：** AI 智能助手模块（自然语言交互/上下文记忆/RAG 知识增强/模块跳转）、思维链分析模块（6 类分析场景/逐步推理链条输出）、案例推理模块（历史案例库/特征相似度检索/经验复用）、知识库管理模块（文档上传/查看/数据库持久化）。

**（五）仿真训练类：** 仿真模拟模块（多实例并行/碰撞检测/地理围栏/快照管理）、强化学习训练模块（Q-Learning/训练参数配置/奖励曲线/策略导出）。

**（六）平台支撑类：** 维护日志模块（保养历史记录/筛选查询）、性能分析模块（多维度指标/可视化图表）、消息通知中心模块（统一通知管理/类型筛选/批量已读）、并发任务监控模块（Worker Pool 状态/手动任务提交）、系统仪表盘模块（全局统计/模块导航）、数据持久化模块（全量数据库存储/消除文件依赖）。

#### 3.2.8 模块间的协作流程

本系统各模块之间通过明确的数据流与事件驱动机制实现高效协作：

**协作流程 1 —— 飞行任务全链路。** 用户在飞行任务模块创建任务 → 航线规划引擎调用 LLM + NFZ 生成航线 → 禁飞区模块提供地理围栏约束 → GPS 模块实时追踪执行进度 → 电池模块监控能源状态 → 告警模块在异常时触发预警 → 消息通知中心推送任务状态变更 → 性能分析模块记录任务指标。

**协作流程 2 —— 智能决策闭环。** AI 助手接收用户自然语言查询 → RAG 引擎从知识库检索相关文档 → CBR 引擎检索历史相似案例 → CoT 模块执行思维链推理分析 → 结果通过 WebSocket 实时推送至前端 → 用户反馈用于优化知识库与案例库。

**协作流程 3 —— 仿真训练链路。** 仿真模块创建多实例场景 → 强化学习模块在仿真环境中执行训练 → GPS/电池模块模拟遥测数据 → 碰撞检测与禁飞区模块提供安全约束 → 训练完成后策略导出至数据库 → 可加载策略应用于实机飞行任务。

**协作流程 4 —— 数据持久化链路。** 各业务模块产生的数据（遥测、告警、任务、日志等）通过 WriteBatcher 批量聚合 → TaskPool 调度写入任务 → DB Wrapper 执行持久化 → 备份模块定期创建全量快照 → 回滚模块支持任意时刻数据恢复。

## 四、功能实现

### 4.1 功能概述

本系统围绕无人机集群智能管控场景，构建了覆盖"感知—管控—分析—决策—仿真—训练"全链路的 20 个功能模块。系统功能按层次可划分为以下几个维度：

**基础感知与管控功能：** 实现无人机设备的自动发现与统一管理、GPS 实时定位与轨迹追踪、电池多参数监控与阈值告警、飞行任务全生命周期管理，形成对无人机集群的基本态势感知能力。

**安全保障功能：** 提供禁飞区地理围栏管控与 Ray-Casting 实时碰撞检测、多级别告警管理与处理、数据库全量备份与一键回滚恢复，确保飞行安全与数据安全。

**远程接入功能：** 集成 SSH 终端、VNC 桌面（noVNC）与 RDP 远程桌面三种远程接入方式，支持对已注册远程设备的远程调试与操作。

**智能决策功能：** 融合 LLM 大语言模型智能航线规划、BM25 RAG 知识检索增强、CoT 思维链可解释分析、CBR 案例推理经验复用与 AI 智能助手自然语言交互，形成多智能技术协同的决策辅助体系。

**仿真训练功能：** 提供多实例并行仿真引擎（支持 200 台无人机、碰撞检测、地理围栏）与 Q-Learning 强化学习训练模块，构建从仿真验证到策略优化的闭环训练能力。

**平台支撑功能：** 包括统一仪表盘导航、消息通知中心、并发任务监控、性能分析、维护日志、数据持久化等基础平台能力，保障系统的可运维性与可观测性。

### 4.2 功能实现细节

#### 4.2.1 智能航线规划功能

智能航线规划是本系统的核心差异化功能之一，采用 LLM + NFZ 双引擎协同架构实现。

**输入参数：** 起点经纬度、终点经纬度、飞行高度（米）、飞行速度（m/s）、用户自然语言描述的特殊要求（如"避开居民区""沿河道飞行"）。

**LLM 推理阶段：** 系统将上述参数组装为结构化 Prompt，包含系统角色定义（"你是专业的无人机航线规划师"）、输出格式约束（JSON 航点数组，每个航点包含 lat、lng、alt 字段）、安全飞行规则（最大倾斜角、最大速度、电量预留要求）。调用 LLM API 后解析返回的 JSON 航点序列。系统内置 RPM 令牌桶限流机制，防止高频调用超出 API 配额。

**NFZ 纠偏阶段：** 获取数据库中所有已启用的禁飞区（包含中心点、半径或多边形顶点坐标）。对航线的每一段（相邻两个航点之间的线段）执行 Ray-Casting 碰撞检测。若某段航线穿越禁飞区，计算禁飞区边界上最近的两个切点，插入绕飞航点使航线绕过禁飞区外围。

**轨迹平滑阶段：** 对纠偏后的航点序列应用三次贝塞尔曲线（Cubic Bézier）插值算法。以每两个相邻航点为控制点，生成平滑的曲线轨迹。平滑后的轨迹在保证不穿越禁飞区的前提下，消除急转弯航段，提升飞行器执行的舒适性与能源效率。

**输出结果：** 最终航线包含航点列表（经纬度、高度）、总飞行距离（km）、预计飞行时间（分钟）、途经禁飞区绕飞信息。结果通过 RESTful API 返回前端，在 Leaflet 地图上以折线 + 航点标记形式可视化展示。

#### 4.2.2 仿真模拟引擎功能

仿真模拟引擎是系统进行策略验证与强化学习训练的核心基础设施。

**场景配置：** 用户可配置仿真参数，包括无人机数量（1~200 台）、各无人机的初始位置与目标位置、电池容量与初始电量、飞行速度、禁飞区列表、仿真时间步长（默认 1 秒）与最大仿真时长。

**Tick 驱动引擎：** 仿真采用离散时间步（Tick）驱动模式。每个 Tick 内，引擎按序执行以下操作：根据当前速度向量更新每架无人机的经纬度与高度；按固定消耗率扣减电池电量；遍历所有无人机对，计算欧氏距离并检测碰撞风险；检测每架无人机当前位置是否位于禁飞区内；判断是否到达目标位置；检测是否触发低电量返航阈值。

**碰撞避让机制：** 当两架无人机间距小于安全阈值（默认 50 米），触发避让策略：优先级较低的无人机自动偏转航向 30°（左偏或右偏，取冲突方向的反向），同时提升高度 10 米。避让持续至安全距离恢复后恢复原航向。碰撞事件记录至仿真日志。

**地理围栏校验：** 每个 Tick 检测所有无人机是否进入禁飞区。若检测到违规，立即触发紧急偏转并记录违规事件。禁飞区判定采用 Ray-Casting 点在多边形内检测算法。

**实时状态广播：** 每个 Tick 结束后，引擎将所有无人机的最新状态（位置、电量、状态码、速度向量）封装为 JSON 消息，通过 WebSocket 广播至前端。前端 Leaflet 地图实时更新无人机标记位置与状态颜色编码。

#### 4.2.3 AI 智能助手功能

AI 智能助手是系统面向操作人员的自然语言交互入口，集成多项智能能力。

**自然语言交互：** 用户通过浮窗式聊天界面输入自然语言问题或指令。助手支持多轮对话上下文记忆（基于会话 ID 维护对话历史），能够理解上下文语境并给出连贯回答。

**RAG 知识增强：** 当用户查询涉及领域专业知识时，助手自动触发 RAG 检索流程。BM25 引擎从知识库中检索 Top-3 相关文档片段，作为上下文注入 LLM Prompt，确保回答基于权威领域知识而非模型幻觉。

**CoT 推理分析：** 助手支持调用 CoT 思维链推理能力，对复杂问题进行逐步分析。推理结果以结构化格式呈现（推理步骤、依据、结论、置信度），增强回答的可解释性与可信度。

**模块快速跳转：** 助手能够识别用户意图中涉及的功能模块，并生成可点击的模块链接。例如用户询问"查看 3 号无人机电池状态"，助手在回答中嵌入跳转至电池监控模块的快捷链接。

**前端交互设计：** 聊天浮窗支持拖拽移动与位置持久化（localStorage），支持代码高亮渲染（Markdown 格式回答），消息气泡区分用户与助手消息，底部输入框支持 Enter 发送与 Shift+Enter 换行。

### 4.3 附加功能扩展

在核心功能之外，系统还提供以下扩展功能以增强平台的实用性与完整性：

**（1）消息通知中心。** 统一管理系统内所有通知消息，包括任务状态变更、告警事件、系统公告等。支持通知类型筛选（全部/未读/按类型）、批量标记已读、通知铃铛实时角标提醒。通知数据通过 Redis 缓存加速查询，WebSocket 实时推送新通知事件。离线状态（WebSocket 未连接）时自动抑制通知推送，避免无效消息积压。

**（2）并发任务监控。** 提供 TaskPool 运行时状态的实时可视化面板，展示 IO Pool 与 CPU Pool 的队列深度、活跃 Worker 数、已完成任务数与失败任务数。支持手动提交测试任务（指定优先级与执行类型）以验证调度器行为。

**（3）性能分析与统计。** 多维度性能指标聚合分析，包括单机/全局飞行时长统计、任务完成率、电池健康度趋势、GPS 信号质量分布等。统计数据通过 StatsCache 异步定时刷新（避免每次请求重复计算），Chart.js 图表支持时间范围筛选与数据导出。

**（4）维护日志管理。** 记录每架无人机的维护保养历史（维护类型、维护人员、维护内容、维护时间），支持按设备/时间/类型多条件筛选与分页查询，为设备全生命周期管理提供数据支撑。

### 4.4 测试与优化

**（1）并发压测。** 系统并发框架（TaskPool）已通过 9 项单元测试用例，涵盖正常提交、优先级调度、批量执行、节流控制、池满拒绝等场景。其中含 5000 并发任务提交压力测试，验证了队列容量 1024 条件下的调度稳定性与 Worker 资源回收能力。

**（2）功能回归测试。** 所有 20 个功能模块均已通过手动功能验证，包括 CRUD 操作、分页查询、WebSocket 推送、地图交互、图表渲染等核心交互场景。数据库迁移脚本支持 34 张表的自动创建与增量更新，经反复迭代验证无 DDL 冲突。

**（3）性能优化措施。** WriteBatcher 批量写入将 GPS 高频数据的数据库写入频率降低约 80%；ThrottledBroadcast 节流器将 WebSocket 推送频率控制在合理范围内，避免前端消息堆积；StatsCache 异步刷新将统计接口响应时间从数百毫秒级降至个位数毫秒级；Redis 缓存命中率显著提升高频查询接口的响应速度。

**（4）稳定性优化。** Gin Panic Recovery 中间件确保单个请求异常不会导致服务进程崩溃；Redis 降级策略确保缓存服务不可用时核心功能正常运行；LLM 降级策略确保 AI 服务不可用时航线规划自动回退为直线方案。

## 五、系统数据介绍

### 5.1 系统核心数据定义

本系统数据模型以**无人机实体（drones）**为中心节点，各类业务数据通过 `drone_id` 外键关联，形成星型数据关系网络。系统共计 34 张数据库表，按数据职能划分为以下四类：

**感知遥测类：**

- **drones 表：** 无人机设备信息（名称、机型、Agent 标识、运行状态等），在线/离线状态由 TCP Gateway 心跳自动判定。
- **gps_data 表：** GPS 定位数据（经纬度、高度、速度、航向角、卫星数等），高频写入，通过 WriteBatcher 批量持久化。
- **battery_data 表：** 电池监测数据（电压、电流、电量、温度、循环次数等），高频写入，独立 WriteBatcher 聚合。

**业务管控类：**

- **flight_missions 表：** 飞行任务数据（起终点坐标、飞行参数、状态流转、航线 JSON），支持全生命周期管理。
- **no_fly_zones 表：** 禁飞区地理围栏（圆形半径/多边形顶点坐标、启用状态），为 NFZ 碰撞检测提供约束。
- **alerts 表：** 多级别告警记录（信息/警告/严重/紧急），含确认状态与处理措施审计链路。

**智能决策类：**

- **knowledge_docs 表：** RAG 知识库文档（标题、内容、文件名），启动时加载并构建 BM25 索引。
- **data_store 表：** 通用键值存储，承载 RL 策略 JSON 与仿真快照 JSON。

**平台支撑类：**

- **backup_records 表：** 全量备份 SQL Dump 持久化至数据库，无本地文件依赖。
- **users、maintenance_logs、notifications、ai_conversations、cot_analyses、cbr_cases、sim_configs、rl_training_logs** 等辅助业务表，覆盖用户管理、维护日志、消息通知、AI 对话、CoT 分析、案例推理、仿真配置与 RL 训练等业务场景。

### 5.2 系统用例说明

**用例 1 —— 飞行任务创建与执行。** 用户登录系统后，进入飞行任务模块，通过两步向导创建飞行任务：第一步填写基础参数（选择无人机、设定起终点、配置飞行参数），第二步调用智能航线规划生成航线并确认。任务创建后进入"待执行"状态，用户点击"开始执行"后任务转为"执行中"，GPS 模块实时追踪飞行轨迹，电池模块监控能源状态。任务完成后自动转为"已完成"，性能分析模块记录本次飞行的各项指标。

**用例 2 —— 异常告警处理。** 电池模块检测到某架无人机电量低于 20% 阈值，自动生成"警告"级别告警并通过 WebSocket 推送至前端。用户在告警管理面板看到未处理告警，点击确认并录入处理措施（如"指令返航"）。告警状态从"未确认"更新为"已处理"，完整处理记录留存于数据库供事后审计。

**用例 3 —— AI 辅助决策。** 用户遇到复杂飞行场景（如多禁飞区交叉区域），打开 AI 智能助手，输入"请分析从 A 点到 B 点经过多个禁飞区的最优航线方案"。助手自动调用 RAG 引擎检索相关知识文档，结合 CBR 引擎匹配历史类似案例，通过 CoT 推理逐步分析各候选方案的优劣，最终给出带有推理过程与置信度的建议方案。

**用例 4 —— 仿真训练与策略优化。** 用户在仿真模块创建包含 50 架无人机的仿真场景，配置多个禁飞区与复杂目标分布。启动仿真后，强化学习模块在仿真环境中执行 Q-Learning 训练，历经 1000 个 Episode 后策略收敛。用户查看奖励曲线确认训练效果，导出策略 JSON 文件，后续可加载至实机飞行任务中应用。

**用例 5 —— 数据备份与灾难恢复。** 用户定期在备份模块点击"创建备份"，系统自动执行数据库全量 SQL Dump 并将 SQL 内容存储至 `backup_records.sql_content` 字段。若系统发生数据异常，用户从备份列表选择目标备份点击"恢复"，系统自动在恢复前创建当前状态的安全备份，随后执行 SQL 恢复操作，实现数据回滚。

## 六、通信协议设计和接口定义

### 6.1 通信协议设计

本系统涉及多层通信链路，各层通信协议设计如下：

**（1）无人机与云端通信协议。** 无人机飞控系统通过 MAVLink v2 协议与云端服务器通信。MAVLink 是轻量级的无人系统通信协议，采用二进制序列化格式，单条消息长度 8~263 字节，支持消息确认与重传机制。上行消息包括心跳（HEARTBEAT）、GPS 位置（GLOBAL_POSITION_INT）、电池状态（BATTERY_STATUS）、飞行模式（FLIGHT_MODE）等。下行消息包括航线下发（MISSION_ITEM）、模式切换（SET_MODE）、紧急返航（RTL）等控制指令。通信链路基于 4G 蜂窝网络 TCP 长连接承载。后端 TCP Gateway 根据每路 4G 连接中 MAVLink HEARTBEAT 消息携带的 system_id 自动识别不同无人机，并将其与已发现的无人机条目进行关联，从而统计当前在线发送数据的无人机数量。

**（2）前端与后端通信协议。** 前端与后端之间采用两种通信协议并行工作：

- **HTTP/HTTPS RESTful API：** 用于请求-响应式交互，包括资源的 CRUD 操作、查询检索、配置管理等。所有 API 遵循统一的 URL 路径规范（`/api/v1/{resource}`）、统一的请求参数格式（分页：`page`/`page_size`，排序：`sort_by`/`sort_order`）与统一的 JSON 响应格式（`{"code": 200, "data": {...}, "message": "success"}`）。
- **WebSocket 长连接：** 用于服务端主动推送实时数据，包括 GPS 位置更新、电池状态变化、任务状态流转、告警事件通知、仿真状态广播等。WebSocket 端点为 `/ws`，消息格式为 JSON（`{"type": "event_name", "data": {...}}`）。前端通过事件类型字段进行消息路由与分发。

**（3）后端与数据库通信协议。** 后端通过 Go 标准库 `database/sql` 接口与数据库通信，底层使用 MySQL 驱动（`go-sql-driver/mysql`）或 SQLite 驱动（`modernc.org/sqlite`）。连接池参数：最大打开连接数 25，最大空闲连接数 10，连接最大存活时间 5 分钟。

**（4）后端与 Redis 通信协议。** 后端通过 `go-redis/redis` 客户端库与 Redis 实例通信，采用 RESP（Redis Serialization Protocol）协议。主要使用命令包括 GET/SET（会话缓存）、INCR/EXPIRE（分布式限流计数器）、HGET/HSET（统计数据缓存）、DEL（缓存失效）。

**（5）后端与 LLM API 通信协议。** 后端通过 HTTPS 调用外部 LLM API（兼容 OpenAI API 格式），请求体为 JSON 格式（包含 model、messages、temperature 等参数），响应体为 JSON 格式（包含 choices、usage 等字段）。内置 RPM 令牌桶限流机制控制调用频率。

### 6.2 接口定义

系统前后端交互接口按功能模块划分如下：

**（1）用户认证接口。**

- `POST /api/v1/register`：用户注册，参数：username、password，返回：user_id。
- `POST /api/v1/login`：用户登录，参数：username、password，返回：token、expires_at。
- `GET /api/v1/me`：获取当前用户信息（需 Token 鉴权），返回：user_id、username。

**（2）无人机管理接口。**

- `GET /api/v1/drones`：分页查询无人机列表，参数：page、page_size、status（可选），返回：列表 + 总数。
- `POST /api/v1/drones`：系统自动发现时创建无人机记录（由 TCP Gateway 内部调用，前端无需手动触发）。
- `GET /api/v1/drones/:id`：查询单台无人机详情，返回：完整属性。
- `PUT /api/v1/drones/:id`：更新无人机信息，参数：name、model、status，返回：更新后数据。
- `DELETE /api/v1/drones/:id`：删除无人机，返回：操作结果。

**（3）GPS 数据接口。**

- `POST /api/v1/gps`：上报 GPS 数据（高频），参数：drone_id、latitude、longitude、altitude、speed 等。
- `GET /api/v1/gps/:drone_id`：查询指定无人机 GPS 历史，参数：page、page_size、start_time、end_time。
- `GET /api/v1/gps/:drone_id/latest`：查询指定无人机最新 GPS 位置。

**（4）电池数据接口。**

- `POST /api/v1/battery`：上报电池数据（高频），参数：drone_id、voltage、current、percentage、temperature。
- `GET /api/v1/battery/:drone_id`：查询指定无人机电池历史记录。

**（5）飞行任务接口。**

- `GET /api/v1/flights`：分页查询飞行任务列表。

## 七、测试与验证方案

### 7.1 单元测试

系统关键模块均配备独立的单元测试用例，确保核心功能的正确性与稳定性。

**（1）并发框架单元测试（taskpool_test.go）。** 共 9 项测试用例，覆盖以下场景：

- **TestBasicSubmit：** 验证基本任务提交与执行，确认任务在提交后被正确调度并执行完成。
- **TestPriorityScheduling：** 验证 4 级优先级队列的调度顺序，确认 Critical 优先级任务优先于 Low 优先级任务执行。
- **TestIOPoolSubmit：** 验证 IO 密集型任务被正确分配至 IO Worker Pool 执行。
- **TestCPUPoolSubmit：** 验证 CPU 密集型任务被正确分配至 CPU Worker Pool 执行。
- **TestBatchExecutor：** 验证 WriteBatcher 批量聚合与定时触发机制，确认高频写入请求被正确聚合为批次执行。
- **TestThrottler：** 验证节流器在指定时间窗口内仅允许一条消息通过，超出窗口的消息被丢弃。
- **TestPoolFull：** 验证队列满时新提交的任务返回错误（而非阻塞），确保系统在高负载下不会死锁。
- **TestGracefulShutdown：** 验证优雅关闭流程，确认 Shutdown 后不再接受新任务，已提交任务执行完毕后 Worker 正常退出。
- **TestConcurrentSubmit5000：** 5000 并发任务提交压力测试，验证在极端并发条件下调度器的稳定性与资源回收能力。

**（2）数据库层单元测试。** 验证 AdaptSQL 函数在 SQLite 与 MySQL 模式下的 SQL 方言转换正确性。测试用例包括：`INSERT OR REPLACE` → `REPLACE INTO` 转换、`AUTOINCREMENT` → `AUTO_INCREMENT` 转换、`DATETIME('now')` → `NOW()` 转换等。

**（3）NFZ 碰撞检测单元测试。** 验证 Ray-Casting 算法对多种几何场景的正确性：点在多边形内部、点在多边形外部、点在多边形边上、线段与多边形相交、线段不与多边形相交。

**（4）BM25 检索单元测试。** 验证 BM25 评分函数的计算正确性，以及 Top-K 检索结果排序的准确性。测试用例包括：单文档匹配、多文档排序、空查询处理、中文分词准确性。

### 7.2 集成测试场景

系统集成测试覆盖端到端的业务流程验证：

**场景 1 —— 用户认证全流程。** 注册新用户 → 登录获取 Token → 使用 Token 访问受保护接口 → Token 过期后访问返回 401 → 重新登录获取新 Token。验证认证链路的完整性与安全性。

**场景 2 —— 飞行任务全生命周期。** 无人机自动发现上线 → 创建飞行任务 → 调用航线规划 → 启动任务 → 模拟 GPS 数据上报 → 模拟电池数据上报 → 验证 WebSocket 推送 → 任务完成 → 查询性能统计。验证多模块协同工作的正确性。

**场景 3 —— 仿真训练闭环。** 创建仿真场景（10 架无人机 + 3 个禁飞区）→ 启动仿真 → 验证 Tick 循环正常推进 → 验证碰撞检测触发 → 验证禁飞区违规检测 → 创建快照 → 停止仿真 → 从快照恢复 → 启动 RL 训练 → 验证奖励收敛 → 导出策略。验证仿真与强化学习链路的端到端正确性。

**场景 4 —— AI 智能助手集成。** 上传知识文档至知识库 → 发送包含领域问题的查询 → 验证 RAG 引擎返回相关文档引用 → 验证 LLM 回答基于检索文档内容 → 触发 CoT 分析 → 验证推理链条格式正确。验证智能决策链路的集成正确性。

**场景 5 —— 备份恢复完整性。** 创建测试数据（无人机、GPS、任务等）→ 执行全量备份 → 删除部分测试数据 → 执行恢复 → 验证恢复前自动创建安全备份 → 验证数据完整恢复至备份时刻的状态。

**场景 6 —— 高并发 WebSocket 推送。** 建立 50 个 WebSocket 连接 → 以 10ms 间隔触发 GPS 数据更新 → 验证 ThrottledBroadcast 将推送频率控制在 200ms → 断开 25 个连接 → 验证 Hub 正确注销断开的 Client → 验证剩余连接正常接收推送。

### 7.3 可靠性设计

系统在多个层面实施可靠性保障机制：

**（1）服务进程级可靠性。** Gin 框架内置 Panic Recovery 中间件，当任何 HTTP Handler 发生未捕获的 panic 时，中间件自动恢复并返回 500 Internal Server Error，避免整个服务进程崩溃。Recovery 日志记录完整的调用栈信息，便于事后排查。

**（2）缓存层降级可靠性。** Redis 连接失败时，系统自动降级为纯内存模式运行。限流计数器切换为进程内原子计数器；统计缓存切换为内存 Map；通知缓存直接查询数据库。降级过程对前端完全透明，不影响功能可用性，仅在性能上有所下降。

**（3）AI 服务降级可靠性。** LLM API 不可用时（网络超时、配额耗尽、API 错误），航线规划自动降级为直线方案（起点到终点的直连航线），AI 助手返回友好的降级提示信息。降级策略确保核心飞行管控功能不受 AI 服务可用性影响。

**（4）数据持久化可靠性。** 全量数据存储于关系型数据库，消除对本地文件系统的依赖。WriteBatcher 在定时 Flush 时若数据库写入失败，将失败数据保留在缓冲区并在下一个 Flush 周期重试。备份恢复前自动创建安全备份，防止恢复操作导致数据丢失。

**（5）通信链路可靠性。** WebSocket 连接实现自动重连机制（前端指数退避策略，初始 1 秒，最大 30 秒）。Hub 对每个 Client 维护写入超时检测，超时后自动注销并释放资源，防止僵尸连接占用服务器资源。4G 通信链路支持心跳保活与断线重连。

**（6）并发安全可靠性。** TaskPool 内部使用 sync.Mutex 与 channel 保障并发安全。队列容量 1024 条件下，池满时 Submit 操作返回错误而非阻塞，避免调用方 goroutine 泄漏。Shutdown 流程先关闭任务接收通道，等待所有 Worker 完成当前任务后退出，保证数据一致性。

**（7）安全审计可靠性。** 所有关键操作（用户登录、任务创建/启停、告警确认、备份/恢复、AI 交互）均记录完整的操作日志（操作人、操作时间、操作内容、操作结果），存储于数据库，支持事后审计追溯。API 限流机制（3000 次/分钟/IP）有效防止恶意请求对系统可用性的影响。
