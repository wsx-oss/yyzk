# 并发任务监控 功能介绍

## 一、功能概述

并发任务监控是"CloudControl"的基础设施模块，为整个系统提供**统一的后台任务调度、执行和可观测能力**。通过该模块，用户可以实时查看后端任务池的运行状态、各任务组的执行指标、Worker 利用率和队列压力，确保系统后台服务稳定高效运行。

---

## 二、设计背景

### 2.1 为什么需要统一并发框架？

在"CloudControl"的运行过程中，后端存在大量需要周期性或异步执行的后台任务：

1. **系统监控类**：阈值告警检测（CPU/内存/磁盘）、设备离线检测（GPS → 无人机 → 硬件级联）
2. **AI 巡检类**：电池低电量检测、无人机离线检测、严重告警检测、硬件异常检测、错误日志检测、任务完成通知
3. **高频数据类**：GPS 位置推送、电池状态推送（来自 Agent 的高频写入和 WebSocket 广播）
4. **计算密集类**：航线规划计算、CoT 思维链分析、统计数据聚合

在引入统一并发框架之前，这些任务各自使用 `go func() { for { ticker } }` 模式独立运行，存在以下问题：
- 无法统一监控各任务的执行状态和健康度
- 缺乏超时控制，单个任务卡死可能影响系统稳定
- 缺乏 panic 恢复机制，一个任务崩溃可能拖垮整个进程
- 高频数据库写入和 WebSocket 推送造成资源浪费

### 2.2 解决方案

引入统一 Worker Pool（任务池）架构，所有后台任务通过任务池统一调度和执行，同时提供批量写入、推送节流、统计缓存等辅助工具。

---

## 三、技术实现

### 3.1 Worker Pool 引擎（`internal/taskpool/pool.go`）

核心组件，提供以下能力：

| 能力 | 实现方式 |
|------|----------|
| **双池隔离** | IO 池（16 workers）处理数据库查询、网络请求等 I/O 密集任务；CPU 池（4 workers）处理航线计算等 CPU 密集任务 |
| **优先级调度** | 4 级优先级：Realtime（0）> High（1）> Normal（2）> Low（3），高优先级任务优先进入 worker |
| **超时控制** | 每个任务独立 `context.WithTimeout`，默认 30 秒，可按任务自定义 |
| **Panic 恢复** | `defer recover()` 捕获 panic，记录堆栈信息，recovered 计数器递增，不影响其他任务 |
| **周期调度** | `SchedulePeriodic(name, group, interval, priority, fn)` 注册定时任务，支持命名替换和取消 |
| **批量执行** | `RunBatch(tasks)` 提交一组任务并等待全部完成，用于 fan-out 场景 |
| **指标采集** | 全局 + 按任务组采集：提交数/完成数/失败数/恢复数/平均耗时/队列长度/活跃 worker 数 |

### 3.2 辅助并发工具（`internal/taskpool/batcher.go`）

| 工具 | 用途 | 参数 |
|------|------|------|
| `WriteBatcher` | 将高频数据库写入聚合为批次，减少 DB 连接压力 | GPS 历史：50 条或 2 秒 flush；电池记录：30 条或 3 秒 flush |
| `Throttler` | 对高频 WebSocket 推送进行节流，避免客户端被刷爆 | GPS：200ms 窗口；电池：500ms 窗口 |
| `StatsCache` | 将耗时统计查询改为后台定时刷新 + 请求读缓存 | 6 类缓存（drones/gps/battery/alerts/hardware/flight_missions），10-15 秒刷新 |

### 3.3 整合层（`internal/handlers/pool_integration.go`）

提供业务层与底层框架之间的桥接：

- `BatchGPSHistory(db, deviceID, lat, lng, alt, speed, heading)` — GPS 历史批写入口
- `BatchBatteryRecord(db, ...)` — 电池记录批写入口
- `ThrottledBroadcast(topic, event)` — 按 topic 节流的 WebSocket 广播
- `InitStatsCaches(db)` / `StopStatsCaches()` — 统计缓存生命周期管理
- `SubmitSimulationTask(pool, droneID, fn)` — 模拟任务提交
- `SubmitFlightPlanTask(pool, planID, fn)` — 航线计算任务提交

### 3.4 已接入的后台任务

| 任务名 | 任务组 | 间隔 | 说明 |
|--------|--------|------|------|
| `threshold:alerts` | monitoring | 10s | CPU/内存/磁盘阈值检测，带 cooldown 防重复告警 |
| `offline:detection` | monitoring | 10s | GPS 设备 → 无人机 → 硬件级联离线检测 |
| `patrol:battery` | patrol | 30s | 低电量（<20%）+ 高温（>50°C）检测 |
| `patrol:drones` | patrol | 30s | 无人机离线事件检测 |
| `patrol:alerts` | patrol | 60s | 新增严重告警检测 |
| `patrol:hardware` | patrol | 60s | 硬件离线 + 资源异常检测 |
| `patrol:logs` | patrol | 120s | 错误级别日志检测 |
| `patrol:missions` | patrol | 30s | 已完成任务通知 |

### 3.5 优雅关闭

服务关闭时按以下顺序执行，确保数据不丢失：

```
SIGINT/SIGTERM
  → StopBatchers()        # flush 所有写入缓冲
  → StopStatsCaches()     # 停止缓存刷新
  → pool.Shutdown(5s)     # 等待所有 worker 退出
  → srv.Shutdown(5s)      # 关闭 HTTP 服务器
```

## 五、前端页面

路径：dashboard 侧边栏 →「并发任务监控」（`web/modules/concurrency.html`）

### 5.1 页面组成

| 区域 | 内容 |
|------|------|
| **汇总卡片** | 已提交任务、已完成、失败、排队中、平均耗时(ms)、异常恢复 |
| **Worker 池状态** | IO Workers / CPU Workers / IO 队列 / CPU 队列的进度条和数值 |
| **趋势折线图** | 每轮采样的完成/失败增量（滑动窗口 30 个数据点） |
| **任务组环形图** | 各任务组（系统监控/AI巡检/无人机模拟/航线规划/CoT分析）的任务量占比 |
| **任务组详情表** | 按组展示：已提交、已完成、失败、成功率、平均耗时 |

### 5.2 自动刷新

支持 4 种刷新间隔：2 秒 / 5 秒（默认）/ 10 秒 / 关闭，以及手动「立即刷新」按钮。

## 六、关键设计决策

### 6.1 节流 vs 直接广播

- **高频 Agent 推送**（GPS 位置、电池状态）→ 使用 `ThrottledBroadcast` 节流，窗口内只保留最新数据
- **离散用户操作**（任务创建/删除/更新、告警确认）→ 保留 `hub.Broadcast` 直接广播，确保每条事件都送达

### 6.2 批写 vs 实时写

- **GPS 历史轨迹**：允许最多 50 条或 2 秒延迟，换取显著减少的 DB 连接压力
- **主表实时字段**（如 gps_devices.last_update）：仍然实时写入，确保离线检测准确
- **进程崩溃**：最多丢失一个批次的历史点，主表数据不受影响

### 6.3 统计缓存延迟

- `/api/stats/cached` 返回的数据最旧为 15 秒前的快照
- 适用于仪表盘展示，不适用于需要毫秒级精度的告警判断（告警仍使用实时查询）
