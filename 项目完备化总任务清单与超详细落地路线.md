## 0. 文档目的与使用方式

### 0.1 你接下来要做什么

你现在的任务不是“再写一份介绍文档”，而是把当前项目从：

- 前端 10 大模块里大多数页面仍是静态展示
- 后端已有不少 API 能力，但缺少模块页接线、缺少历史数据、缺少任务引擎、缺少视频/无人机接入通道

- 10 大模块全部变成“真功能”（可跑通、可演示、可验收）

### 0.2 本文档怎么用（强烈建议按此执行）

- 每次只拿一个“任务卡”（见第 3 章起）去实现。
- 每完成一个任务卡：
  - 在本地跑通并截图/录屏
  - 写 1-3 行“验收记录”（你可以直接贴在该任务卡下面）

### 1.2 你应当坚持的总体原则（避免走弯路）

- **先接线，再增强**：先把静态页面接到真实 API，完成“动态可用”；再做筛选、导出、分页、美化。
- **先快照再实时**：先实现 `GET snapshot` 渲染；再加 WS 或轮询。
- **先最小闭环再扩展**：每个模块至少完成：
  - 能看到真实数据
  - 能触发一次真实动作（新增/删除/确认/同步/更新）
  - 能看到动作结果写入 DB
- **视频/无人机是最后做的高成本模块**：先把 8 个“后端已具备”的模块做成真功能，快速提升整体完成度。

## 4. 阶段 B：模块逐一“接线成真功能”（超详细任务卡）

> 说明：本阶段原则是“尽量不改后端或只做小改”，先让模块页从静态变动态。

### B-1 任务卡：M10 维护操作日志（modules/logs.html）

- 目标：

  - 列表从后端加载
  - 新增日志能落库并立即在列表出现
  - 支持按关键字过滤（前端过滤或后端参数）
- 涉及文件（你需要改哪里）：

  - 前端：`project/web/modules/logs.html`
  - 前端公共：`project/web/modules/common.js`（复用 `api()`）
  - 后端（一般无需改）：`project/internal/handlers/api.go`（`/api/logs/list`、`/api/logs/append`）
- 详细步骤（照着做）：

  1. 在 `logs.html` 页面加载时调用 `api('/logs/list?page=1&page_size=20')`。
  2. 将返回 JSON 映射到表格行：时间、用户、动作、结果/备注等。
  3. “新增日志”按钮：
     - 弹出表单（标题、内容、级别等字段按后端要求）
     - 提交时 `api('/logs/append', { method:'POST', body: JSON.stringify(payload) })`
  4. 提交成功后：
     - 关闭弹窗
     - 重新调用列表接口刷新
  5. 搜索框：
     - 最快方式：前端对已加载列表做 `includes()` 过滤
     - 更正规：给后端 list 增加 query 参数（如果当前后端支持就直接用）
- 验收标准：

  - 刷新页面后，日志列表仍能正确加载
  - 新增一条日志后，在 SQLite 中能查询到记录（并在页面中出现）
- 常见坑：

  - 忘记携带 token：确保使用 `common.js` 的 `api()`
  - 表格渲染 XSS：渲染文本字段建议使用 `textContent`

  4. 前端 monitor/performance 增加时间筛选器，调用 history 接口画趋势图。
- 验收标准：

## 7. 阶段 E：无人机接入（先模拟后实机）

### E-1 你要达成的最终效果

- 在监控端：
  - 看到无人机遥测（姿态、高度、速度、电量、GPS、模式等）实时刷新
  - 看到指令回执（例如起飞/降落/返航）

### E-2 数据与控制分层（必须做，否则后面会崩）

- 遥测数据层：

  - 传输：WS/MQTT/MAVLink 网关
  - 逻辑：只负责实时上报与存储
- 控制指令层：

  - 传输：HTTP -> 后端 -> 指令通道
  - 逻辑：必须鉴权、记录日志、可追溯、可限流
- 视频层：

  - 单独链路（WebRTC/RTSP/HLS），不要和普通 API 混用

### E-3 从“模拟机”到“无人机”迁移步骤（建议你按这 5 步）
