## 一、主讲人答辩稿（建议 8–10 分钟，含演示串词）

### 0. 开场（0:00–0:30）

各位评委老师好，我是本项目的主讲人。我们团队的项目叫**《智能工具人机交互与远程操控系统》**。

一句话概括：我们做的是一套**面向无人机/边缘设备的综合管控平台**，把“设备接入—实时监控—告警处置—远程操作—任务管理—数据同步—智能规划”串成一个闭环，解决多设备分散管理、状态不可视、控制链路割裂的问题。

---

### 1. 背景与痛点（0:30–1:20）

在实际无人机或边缘设备运维中，常见痛点是：

- 设备数量多、分布散，缺少统一资产管理与状态面板。
- 位置、视频、电池、任务等数据分散在多个工具，信息不互通。
- 运维需要远程桌面/SSH，但传统方案依赖客户端或复杂部署。
- 网络不稳定、设备端资源有限，要求系统**轻量、可离线、可快速部署**。

因此我们希望搭建一个平台，让操作人员在一个 Web 控制台内完成：

- **注册无人机与关联设备**
- **实时 GPS/电池/任务状态**
- **阈值告警与处置**
- **浏览器内远程桌面与 SSH**
- **数据同步与性能报告**
- **LLM 驱动的智能航线规划**（并且在 LLM 不可用时自动降级）

---

### 2. 系统总体方案（1:20–2:30）

我们的方案是“**前后端一体化 + 设备端 Agent + 事件驱动实时推送**”：

- **后端**：Go + Gin 提供 REST API；SQLite 持久化；WebSocket 做实时推送。
- **前端**：原生 HTML/CSS/JS 的静态页面（仪表盘 + 15 个功能模块页，另含登录/注册认证入口），由后端 `embed` 内嵌后统一托管在 `/app`，实现**单可执行文件部署**。
- **设备端**：`hw-agent` 可部署在无人机/目标设备上，支持 Push/Pull 两种模式，把硬件指标、GPS、电池等数据上报到平台。

数据流上：

- 设备端持续上报 → 后端写入 SQLite → 通过 WebSocket 把“变化事件”推送给前端 → 前端实时更新地图/曲线/告警。

---

### 3. 功能闭环展示（2:30–5:30）

我们系统整体按“功能域”可归纳为 **16 大功能**：其中**仪表盘内可切换的业务模块是 15 个**，另外还有**用户认证（注册/登录/会话）**作为系统入口与安全基座。我重点讲“无人机运维闭环”相关的几个核心模块：

#### 3.1 用户认证与安全

- 支持注册/登录/登出，会话 Token 存储在 `sessions` 表，默认 24 小时过期。
- 密码用 bcrypt 哈希保存。
- 可选的 `SC_API_TOKEN` 用于保护后端 API（便于在部署时做二次防护）。

#### 3.2 无人机管理（资产与联动）

- 注册无人机时会**自动创建关联的远程设备（VNC/SSH/RDP）与 GPS 设备**，避免手工多处配置。
- 删除无人机会级联清理关联数据，保证一致性。
- 提供无人机统计：在线/离线、协议分布等。

#### 3.3 GPS/位置信息（实时地图 + 轨迹 + 电子围栏）

- 前端使用 Leaflet + OpenStreetMap 展示实时位置。
- 后端提供 GPS 推送接口与历史轨迹查询；支持 WebSocket 事件 `gps_update`。
- 电子围栏越界触发围栏告警，并在告警模块集中展示。

#### 3.4 电池监控（实时推送 + 自动告警）

- 监控电量/电压/电流/温度/健康度。
- 后端内置阈值规则（例如低电量、严重低电、高温、低健康度），自动写入 `battery_alerts`。
- 采用 WebSocket 事件驱动推送，数据变化即通知前端，降低轮询延迟。

#### 3.5 飞行任务管理（状态机 + 日志 + 实时通知）

- 任务支持创建/编辑/删除，并记录任务日志。
- 飞行阶段采用状态机：**待命→起飞→巡航→执行任务→返航→降落**。
- 任务变更通过 WebSocket 推送，前端能第一时间同步。

#### 3.6 远程控制（浏览器内 VNC/SSH + RDP 文件）

- VNC：浏览器端 noVNC 渲染，后端提供 TCP→WebSocket 代理。
- SSH：浏览器端 xterm.js，后端提供 SSH→WebSocket 代理。
- RDP：生成 `.rdp` 文件供本地客户端连接。

#### 3.7 数据同步引擎（多节点、18 张表白名单）

- 为多设备协同场景提供同步任务管理：创建/启停/进度追踪。
- 同步表采用白名单（18 张核心表），防止误同步敏感或无关数据。
- 支持全量/增量模式，并提供 IP 有效性预检。

#### 3.8 系统监控与告警（平台自身可观测）

- 使用 `gopsutil` 采集 CPU/内存/磁盘/网络。
- WebSocket 推送实时曲线；阈值超限自动写入 `alerts`。

---

### 4. 智能航线规划：LLM + 降级规划（5:30–6:50）

我们的亮点之一是**智能航线规划**：

- 后端集成大模型（OpenAI 兼容接口，默认配置通义千问 `qwen-plus`），输入任务需求与约束，输出结构化航点 JSON。
- 同时接入高德地图 API，支持地理编码/逆地理编码，让任务输入更贴近自然语言地址。
- **关键工程点**是可用性：当 `LLM_API_KEY` 或外部服务不可用时，系统会自动降级为直线规划器，保证核心流程不断。

---

### 5. 工程实现与可部署性（6:50–7:50）

我们从工程交付角度做了几件关键事情：

- **单文件部署**：Go `embed` 把前端静态资源打包进后端二进制，部署不依赖 Nginx/Node 构建链。
- **SQLite 嵌入式**：零配置、单文件存储，适合边缘/机载环境。
- **后台任务**：离线检测、阈值告警、同步任务等通过 goroutine + ticker 运行。
- **在线/离线判定**：以 GPS 上报时间为核心依据，超过阈值自动标记离线，并级联更新无人机与远程设备状态。

## 二、评委可能提问（按“追问链路”准备）

> 说明：每个问题都给出“要点式回答”，方便你现场快速组织语言。

### A. 选型与架构

#### Q1：为什么后端选 Go + Gin？

- **并发能力强**：多设备数据上报与 WebSocket 连接并发高，goroutine 成本低。
- **单文件交付**：Go 编译为单可执行文件，便于在边缘/机载环境部署。
- **生态成熟**：Gin、WebSocket、SQLite 驱动与系统指标采集库完善。

#### Q2：为什么数据库选 SQLite，而不是 MySQL/PostgreSQL？

- **零依赖、零运维**：适合边缘设备与演示环境，避免额外部署数据库服务。
- **单文件易迁移**：备份/同步更简单。
- **配合 WAL**：并发读性能更好，满足本项目数据量与实时性要求。

#### Q3：前端为什么不用 Vue/React？

- **减少构建链复杂度**：不用 Node/npm，便于快速交付与离线部署。
- **配合 embed**：前端静态文件直接编译进后端，部署极简。
- **页面模块化**：15 个模块独立页面，需求更偏管理台而非复杂 SPA。

#### Q4：你们的系统“核心链路”是什么？

- **设备端数据接入**（Agent Push/Pull）
- **后端落库**（SQLite）
- **实时推送**（WebSocket Hub，按 topic 广播）
- **前端可视化**（地图/曲线/列表）
- **告警处置**（阈值/围栏/电池异常 → 记录/确认）
- **远程操作**（VNC/SSH/RDP）

---

### B. 实时性与可靠性

#### Q5：为什么要用 WebSocket？不用轮询可以吗？

- 轮询会带来：
  - **延迟不可控**（例如 1–5s 轮询周期）
  - **无效请求多**（数据没变也在拉）
  - **服务器压力更大**
- WebSocket 是事件驱动：数据变化即推送，实时性更好。

#### Q6：设备在线/离线你们怎么判定？会不会误判？

- 以 `last_update` 为准：一定时间窗口内更新视为在线，超过窗口标记离线。
- 误判风险来自网络抖动；应对方式：
  - 采用合理阈值（秒级到分钟级可配）
  - 可以增加“连续 N 次未上报再离线”的策略（后续优化点）

#### Q7：SQLite 并发写会不会锁死？

- 使用 WAL 模式可提升并发读体验。
- 写入仍是串行，但本系统写入量属于可控级别。
- 工程上可进一步优化：批量写入、异步队列、分表或升级外部数据库（作为扩展路线）。

---

### C. 安全与合规

#### Q8：你们的认证与安全措施有哪些？

- 用户密码 bcrypt 哈希，不存明文。
- 会话 token 持久化并设定过期清理。
- 请求限流中间件，降低暴力破解风险。
- 输入校验与基础清理，减少注入风险。
- 可选 `SC_API_TOKEN` 作为 API 二次防护。

#### Q9：VNC/SSH 在浏览器里做，安全吗？会不会泄露凭证？

- 原理是后端做代理：浏览器与后端 WebSocket 通道，后端与目标机 TCP/SSH。
- 风险点与措施：
  - **传输加密**：生产环境需上 HTTPS/WSS（可通过反向代理或直接 TLS）
  - **访问控制**：只有登录用户可访问远程控制页面与接口
  - **凭证管理**：建议用一次性凭证/密钥托管（后续增强项）

#### Q10：数据同步会不会同步到不该同步的表？

- 我们做了**同步表白名单**（18 张核心表），避免全库同步。
- 同步任务可管理、可停用，并提供预检与进度追踪。

---

### D. 智能规划（LLM）相关追问

#### Q11：LLM 输出不稳定怎么办？如何保证可用？

- 我们采取“**强约束结构化输出 + 解析校验 + 自动降级**”：
  - Prompt 约束输出 JSON 航点结构
  - 解析失败/外部服务不可用时，自动降级到直线规划器
- 所以智能能力是“增强项”，不会阻断主流程。

#### Q12：为什么不用传统路径规划算法（A* / RRT 等）？

- 本项目场景更多是“任务编排与航线草稿生成”，LLM 适合把自然语言任务转换成可执行的结构化草稿。
- 传统算法更依赖完整地图/障碍物/环境建模，本项目以平台型管理为主。
- 后续可以把 LLM 产出的草稿再喂给传统规划做二次优化，这是扩展方向。

#### Q13：LLM 成本和隐私怎么处理？

- 成本：通过开关与降级控制；没配 `LLM_API_KEY` 就不调用。
- 隐私：避免上传敏感信息；生产可切换私有化模型或本地模型（扩展路线）。

---

### E. 工程落地与演示可复现

#### Q14：现场怎么保证能跑起来？最小启动步骤是什么？

- 平台端：
  - 进入 `project/`，执行 `go run .`
  - 浏览器访问 `http://127.0.0.1:8080/`
- 设备端（可选）：
  - 运行 `hw-agent`，使用模拟模式定时 Push，上报 GPS/电池/硬件数据。

#### Q15：你们系统的“创新点”到底是什么？

- **一体化闭环**：从资产到任务到控制再到告警与同步，一套系统串联。
- **单文件部署**：后端+前端 embed 打包，适合边缘场景。
- **事件驱动实时推送**：多模块统一基于 WS Hub。
- **LLM + 降级规划**：增强智能但不牺牲可用性。

#### Q16：你们如何衡量性能或可扩展性？

- 目前通过：
  - 请求限流（按 IP）
  - 列表分页与索引优化
  - 性能分析报告模块提供可视化指标
- 扩展路线：
  - 从 SQLite 迁移到外部数据库
  - WebSocket topic 拆分与分布式推送
  - 设备数据写入队列化

### F. 细节追问（更“刁钻”但常见）

#### Q17：为什么你们的离线阈值是秒级？文档里有 60 秒/15 秒两种说法？

- 解释口径：
  - “在线判定”属于策略参数，项目支持可配置/可调整。
  - 不同模块可采用不同阈值（例如 GPS 更敏感、电池与硬件可略宽松）。
  - 答辩时建议统一说：**离线判定采用可配置时间窗口，默认秒级到分钟级均可**。

#### Q18：如果 WebSocket 断线了会怎样？

- 前端可以自动重连（工程上可做指数退避）。
- 断线期间仍可通过 REST 接口刷新“快照数据”。
- 因此 WS 是“实时增强”，不是唯一数据通道。

#### Q19：告警如何避免刷屏？

- 后端可设置告警冷却时间（同类告警在 cooldown 内只记录一次）。
- 重要告警按 severity 分级（warning/critical）。

#### Q20：远程控制为什么要做成代理，而不是前端直连？

- 浏览器安全限制下无法直接发起原生 TCP 连接。
- 代理模式可统一鉴权、记录审计日志、做访问控制与限流。

## 三、答辩现场“快速背诵版”（30 秒复盘）

- 我们做的是：**无人机/边缘设备的一体化管控平台**。
- 核心闭环：**接入 → 落库 → 实时推送 → 可视化 → 告警 → 远程控制 → 任务管理 → 同步与报告**。
- 关键技术：**Go+Gin+SQLite+WebSocket + noVNC/xterm.js + Agent**。
- 亮点：**单文件部署**、**实时事件驱动**、**LLM 航线规划 + 自动降级**、**同步引擎白名单**。
